## 05/03/2026 - Validação de entradas no banco de dados

### Objetivo do dia
Enquanto estava estudando o curso de databases do CS50 de Harvard, parei pra pensar se, no meu projeto, eu deveria fazer validações para as entradas do usuário da API, por exemplo. Em comparação, lembrei de alguns casos de tratamento de dados inseridos em um formulário, por exemplo. Reparei que inserir uma categoria nova para o cardápio através do DRF não é muito diferente e me perguntei se eu deveria aceitar qualquer entrada de quem adicionar ela no banco de dados.

### O que implementei
- Inicialmente, desenvolvi essa lógica para tratar o self.name
```python
splited_name = self.name.strip().split(" ")
normalized_name = []

for word in splited_name:
    if len(word) > 3: #if one of the words inside the name has more than 3 letters, it should be capitalized.
        normalized_name.append(word.capitalize())
    else: #if not, it is only a compound word and should not be capitalized
        normalized_name.append(word)
self.name = " ".join(normalized_name)
```

- Em seguida, desenvolvi uma lógica para self.description
```python
splited_description = self.description.strip().split(".")
normalized_description = []

for phrase in splited_description:
    normalized_description.append(phrase.strip().capitalize())
self.description = ". ".join(normalized_description)
```

- models.py (usando o model Complement como exemplo)
```python
class Complement(models.Model):
    name = models.CharField(max_length=100, blank=False, null=False)
    price = models.DecimalField(max_digits=10, decimal_places=2, null=True, blank=True)
    description = models.TextField(max_length=300, blank=True, null=True)
    is_active = models.BooleanField(default=True)

    def clean(self):
        normalize_name(self) #criei uma função para cada validação (name e description) para deixar models.py mais limpo
        if self.description: #porque é opcional (blank=True)
            normalize_description(self)
```
### Decisões
- **Por que escolhi fazer o tratamento no clean() ao invés de fazer no save()?**
    A explicação é literal. No começo, me peguei pensando se deveria fazer no save() - e nem sei muito bem o motivo de isso ter acontecido - mas o clean() tá ali justamente para isso: Limpar/Tratar os dados. O nome do método já explica. Não tem porque atribuir uma responsabilidade para o save() que não é dele.

- **Por que dividir a validação de name e description em outro arquivo?**
    Principalmente, para deixar o código de models.py mais limpo. Reparei que os únicos campos que exigiam algum tratamento mais voltado para strings eram name e description. A maioria dos models têm name, alguns tem description e o MenuItem tem description obrigatória. Para esse tratamento, pensei em só alterar o método clean de cada model com a lógica, mas depois de montá-la e passar para cada Model.clean(), vi que estava repetindo muito código. Por isso, decidi colocar elas dentro de validators.py.

    Durante esse processo, pensei se essa seria a melhor solução e me deparei com outra: Criar models abstratos para herança. Eu colocaria o um model Category, por exemplo, como herdeiro desse model abstrato - que teria o método clean com essa lógica. Porém, preferi manter o tratamento em um arquivo separado (menu/validators.py) por ser mais simples e por ter algumas diferenças entre cada model. O model ComplementGroup, por exemplo, tem name mas não tem descrição. O model MenuItem tem description obrigatória e não exige um "if self.description:" (confira a seção "O que implementei: models.py (usando o model Complement como exemplo)" para entender melhor). Essas diferenças me fazem priorizar uma solução mais simples no lugar do model abstrato.

    Em relação aos outros campos, consegui fazer um tratamento através de MinValueValidator, de uma comparação entre um campo A e um campo B dentro de clean() isoladamente ou de outras maneiras que não exigiriam uma lógica mais complexa como os campos de name e description, que são CharFields. Por isso, não exigiram uma função específica da minha parte.
