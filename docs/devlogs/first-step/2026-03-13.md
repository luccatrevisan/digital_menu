## 13/03/2026 - Saindo de SQLite3 para PostgreSQL

### Objetivo do dia
Trocar o banco de dados do projeto para PostgreSQL, garantindo mais performance e concorrência.

### O que implementei
- após configurar postgresql no meu computador:
```sql
CREATE DATABASE 'projeto';
CREATE USER 'nome' WITH PASSWORD 'senha';
GRANT ALL PRIVILEGES ON DATABASE 'projeto' TO 'nome'; 
--não utilizei esses nomes para o banco, user e senha (não sou louco) porque preferi deixar esses valores protegidos no arquivo .env
```

- configurações de bancos de dados em settings.py:
```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": os.getenv("DB_NAME"),
        "USER": os.getenv("DB_USER"),
        "PASSWORD": os.getenv("DB_PASSWORD"),
        "HOST": os.getenv("DB_HOST"), #coloquei HOST no .env porque eventualmente irá mudar e fica mais simples de encontrar
        "PORT": os.getenv("DB_PORT"),
    }
}
```

- importei o pacote psycopg2 para integração entre Python e PostgreSQL, atualizando requirements.txt:

- para a migração, pedi ajuda para inteligência artificial generativa (Claude) e ele me passou esse comando:
```python
python manage.py dumpdata menu --natural-foreign --natural-primary --indent 2 --output backup.json
#pega os dados que eu tenho no app menu, em formato json, com indentação, texto no lugar das chaves estrangeiras e das chaves primárias e coloca em um arquivo backup.json

python manage.py loaddata backup.json #depois de trocar as configurações de bancos de dados em settings.py
```

### Decisões
- **Por que escolhi criar um usuário específico do projeto ao invés de utilizar o superuser padrão do PostgreSQL:**
  Principalmente por questões de segurança. Usar um mesmo usuário para gerenciar um ou mais bancos de dados não faz sentido porque, se a segurança for comprometida, o estrago pode ser maior.

### Desafios
1. Na migração dos dados, tive alguns problemas com a formatação deles ao fazer o comando loaddata backup.json. Inicialmente, aparecia um erro de UnicodeDecode.
   - **Como resolvi:** Tentei resolver "na mão", indo para o arquivo backup.json pelo vscode e conferindo se aparecia UTF-8 na parte inferior direita. Como estava UTF-8 - normalmente - pensei que a solução poderia estar na formatação das linhas e não dos dados em si. Logo ao lado de UTF-8, no vscode, tem CRLF ou LF, que definem como as linhas serão sequenciadas. 
   - **O que aprendi:** A formatação das linhas no arquivo json importam e utilizar o formato LF é mais seguro para importações entre sistemas, apesar de o Windows utilizar o formato CRLF.
