### O que implementei
- MinValueValidator para max_quantity no model ComplementGroup 
```python
class ComplementGroup(models.Model):
    max_quantity = models.IntegerField(blank=False, null=False, validators=[MinValueValidator(1)])
    # apesar da validação no método clean(), MinValueValidator garante que o valor de max_quantity nunca será zero.
```

- MinValueValidator decimal para price no model MenuItem
```python
class MenuItem(models.Model):
    price = models.DecimalField(max_digits=10, decimal_places=2, null=False, blank=False, validators=[MinValueValidator(0.01)]) 
    # adicionei um valor mínimo para price, garantindo que não tenha preço negativo e gratuito
    old_price = models.DecimalField(max_digits=10, decimal_places=2, null=True, blank=True) 
    # não preciso adicionar valor mínimo em old_price porque já tem uma validação no método clean (ele precisa ser maior que price)

    def clean(self):
        if self.old_price is not None and self.old_price <= self.price:
            raise ValidationError("Invalid price.")
```

### Decisões
- **Por que não usar related_name para resolver o problema acima?**
    Usar esse recurso contorna a nomeação dada pelo Django, que segue as regras dele baseada nos relacionamentos. Acredito que valha a pena usar o related_name mas vejo como usar os nomes dados pelo Django pode me ajudar a entender melhor o conceito de relacionamentos no banco de dados. Entendo a lógica do Django para dar os nomes e isso tem me ajudado compreender melhor como os relacionamentos funcionam. Pretendo usar related_name em breve, depois de me aprofundar nos estudos de bancos de dados - principalmente sobre relacionamentos.

- **Por que não relacionar Stock com Complements?**
    Em geral, complementos não precisam de estoque. Usar um campo booleano é o suficiente para marcar que aquele complemento estaria indisponível. Me baseio em situações reais para tomar essa decisão, como poucos pedidos conterem complementos. Como o próprio nome deles já diz, não são itens muito almejados e estoque pode ser redundante.

- **Por que escolhi montar um serializer aninhado com todas as informações do cardápio (MenuItemByCategory):**
    Me baseio em facilitar a vida do frontend. Porém, como não tenho muita experiência com requisição de dados de uma API através do frontend, não sei exatamente como isso funcionaria. Eventualmente vou montar a parte frontend do projeto e aprender como faz utilizando Javascript básico e descobrir. De qualquer maneira, pra mim, com o conhecimento que tenho hoje, faz muito sentido juntar todas as informações e montar o cardápio em apenas uma requisição (por parte do front).

    Fica uma dúvida para o futuro: Por conter vários serializers aninhados, o MenuItemByCategory serializer pode causar lentidão? Pode ser feito de um jeito mais eficiente? 

### Desafios
1. Não conseguia mostrar quantity de Stock dentro do MenuItemWithStockSerializer, apesar de montar o aninhamento de forma aparentemente correta
   - **Como resolvi:** Pesquisei sobre relacionamento OneToOne, considerei utilizar related_name - também fazendo algumas pesquisas -, procurei entender a diferença entre prefetch_related e select_related e entender os parâmetros para passar a um serializer. Todas as referências estão na seção abaixo.
   - **O que aprendi:** Descobri que o Django baseia a nomeação de querysets com base NO RELACIONAMENTO do model. Nesse caso, como o relacionamento entre MenuItem e Stock é de OneToOne, não existe um "set" de objetos. Existe apenas um. Por isso, não precisa declarar source="model_set" como eu fiz no MenuItemByCategorySerializer. Esse foi o problema em me basear num nested serializer que contém um tipo diferente de relacionamento.

2. Tive um problema semelhante entre Complement e ComplementGroup. Não conseguia mostrar os complements relacionados com o Grupo no ComplementByGroupSerializer. No serializer, "complements" sempre continha uma lista vazia independentemente do que eu fizesse.
    - **Como resolvi:** Nesse caso, o problema era mais específico. Tendo resolvido logo antes o Desafio 1 (acima), eu já tava achando que a solução estava em entender o relacionamento entre os modelos. Pesquisei sobre ManyToMany, sobre relacionamentos na documentação do DRF e me baseei muito em testes na mão mesmo. No final das contas, não consegui resolver e preferi jogar pra inteligência artificial, mandando mensagens de erro, prints e partes do código. 
    - **O que aprendi:** Eventualmente, com certa dificuldade em identificar o problema, a IA me ajudou a perceber que o problema estava na usabilidade do Django Admin. Dentro do admin, fui atrás dos complementos que eu tinha criado até aqui e tava tudo certo. Depois, fui atrás do ComplementGroup que eu tinha criado e - pelo menos era o que eu pensava - associado aos complementos. Aparentemente, em relacionamentos ManyToMany no Django Admin, você precisa selecionar todos os itens que estão na lista e, COM ELES SELECIONADOS, você salva. 
    
    De cara achei isso muito contra-intuitivo apesar de ter uma instrução abaixo do campo dos complementos. No último link da seção de referências está a solução que eu optei para tornar a usabilidade do usuário (do admin) mais simples. Bastou adicionar esse filtro nas configurações do admin
