## 05/05/2026 - handle loading and API error states
### Goal
Improve user experience by handling the loading and error states. First, to build the logic for loading, then to treat specified status code errors from the API request, like 401 and 404. 

### Implemented
- Created loading.html
```html
{% load static %}
{% block content %}

<div>
    <h2 id="loadingSpinner">Assando cookies...</h2>
</div>
<!-- Note: this approach was later replaced by inline loading state handling in the main container. -->
{% endblock %}
```
 
- Defined two functions: showLoading() and hideLoading()
```javascript
function showLoading(){
    document.getElementById("loadingSpinner").style.display = "block"
}

// the main goal for both was to put the loading partial before the fetch and to remove it right after

function hideLoading(){
    document.getElementById("loadingSpinner").style.display = "none"
}
```
 
- Error treatment within the scope of apiFetch()
```javascript
async function apiFetch(url){
    try{
        const response = await fetch(url, {
            method : "GET",
            headers : {
                "Content-Type" : "application/json"
            }
        });

        if (!response.ok){
            if (response.status === 404){
                throw new Error("Não foi possível buscar informações do cardápio.")
            }
            if (response.status === 401){
                throw new Error("Usuário não autorizado. Tente fazer login")
            }
            throw new Error(`Erro ao carregar cardápio`);
        }

        const data = await response.json();
        return data;
    }
    catch(error){
        console.error(error);
        throw error
    }
}
```

- Error treatment within the scope of the init() function:
```javascript
async function init(){

    const container = document.getElementById("menu-container")

    try{
        container.innerHTML = `<p>Carregando cardápio...</p>`; //changed the showLoading() and hideLoading() approach to this only line

        const data = await apiFetch(url);

        if (!data || data.length === 0){
            container.innerHTML = `<p>Nenhum item disponível no momento.</p>`;
            return
        }

        renderMenu(data);

    } catch(error){
        container.innerHTML = `
        <p>Erro ao carregar informações do cardápio</p>
        <button onclick="init()"> Tentar novamente</button>`;
    }
}
```
 
### Key Decisions
- **Why I chose to put loading and error messages directly index.html instead of using showLoading() and hideLoading():**
    The two functions solution was a suggested approach that I found during some research for the loading element for the application. It was a simple way to show the user that the menu information was loading. But I had other tasks, related to error treatment and decided to put everything together on the menu-container element from index.html.

    So, instead of splitting loading and error handling into separate functions, I chose to manage all states (loading, success, error) directly inside the container. This simplifies the flow and the UI, since my focus is backend architecture. 

### Challenges
1. Before the changes, when I put showLoading() and hideLoading() on my renderMenu() function, the website kept showing just a few items from the menu and changed the style of the page.
   - **How I solved it:** I read the function and the whole main.js file to understand the best way to use the functions. Instead of setting it on renderMenu(), I called the function on init()
   - **What I learned:** I was declaring both of them on the wrong place. When I put the showLoading function on renderMenu, it was messing the rendering and the function would be doing more than its responsability. By calling both on init(), the code is not only cleaner, but correct. 

2. I had some problems deciding where and how to treat the HTTP request errors like status 401 and 404. For example, since I changed the API view to the AllowAny permission class, was there any need to treat the 401 status? And for the 404 error, where should I put the treatment? How can I show the status of the application for the user for better UX?
   - **How I solved it:** After reading the main.js file, I decided to use my init() function to show the error messages to the user and the apiFetch to catch the specified status code.
   - **What I learned:** Since the response object is within the scope of the apiFetch() function, the cleaner way to build the error treatment logic was to throw errors accordingly to the request status inside the function.
