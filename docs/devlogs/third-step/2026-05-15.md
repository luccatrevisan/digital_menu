## 15/05/2026 - Design order domain models
### Goal
Model the order system structure, defining how users place orders and how items are represented, ensuring consistency with existing MenuItem and pricing logic.

### Implemented
- Model Order
```python
class Order(models.Model):
    class Status(models.TextChoices):
        PENDING = "PENDING", "Pending"
        PREPARING = "PREPARING", "Preparing"
        READY = "READY", "Ready"
        DELIVERING = "DELIVERING", "Delivering"
        CONFIRMED = "CONFIRMED", "Confirmed"
        CANCELLED = "CANCELLED", "Cancelled"

    status = models.CharField(max_length=20, choices=Status.choices, default=Status.PENDING)
    total_price = models.DecimalField(max_digits=10, decimal_places=2, default=0)
    created_at = models.DateTimeField(auto_now_add=True)

    def update_total_price(self):
        total = sum(
            item.subtotal for item in self.items.all()
        )

        self.total_price = total
        self.save(update_fields=["total_price"])

    def __str__(self):
        return f"Order #{self.id} - {self.status}"
```
 
- Model OrderItem
```python
class OrderItem(models.Model):
    order = models.ForeignKey(Order, on_delete=models.CASCADE, related_name="items")
    menu_item = models.ForeignKey(MenuItem, on_delete=models.PROTECT)
    quantity = models.PositiveIntegerField()
    unit_price = models.DecimalField(max_digits=10, decimal_places=2)
    subtotal = models.DecimalField(max_digits=10, decimal_places=2)

    def clean(self):
        if self.unit_price < Decimal("0"):
            raise ValidationError(
                {"unit_price": "Unit price cannot be negative."}
            )


    def save(self, *args, **kwargs):
        self.unit_price = self.menu_item.price
        self.subtotal = self.unit_price * self.quantity

        self.full_clean()
        super().save(*args, **kwargs)


    def __str__(self):
        return f"{self.quantity}x {self.menu_item.name}"
```

### Key Decisions
- **Why I store unit_price and subtotal directly in OrderItem:**
    I wanted to preserve historical pricing consistency. If a MenuItem price changes later, previous orders must keep their original values instead of recalculating using the current menu price.

- **Why I chose on_delete=PROTECT in OrderItem:**
    I wanted to preserve order history integrity. Preventing MenuItem deletion avoids breaking historical orders that reference products already purchased by customers.

- **Why subtotal is calculated automatically inside save():**
    I wanted the backend to be the single source of truth for pricing calculations, avoiding reliance on frontend-provided values.

- **Why I used TextChoices for order status:**
    Using TextChoices provides a controlled and standardized set of states, improving readability, maintainability, and reducing invalid status values.

- **Why I decided not to support editable orders:**
    My business operation is time-sensitive and very variable. In some cases, like when the oven is already on high temperature, products are prepared immediately after the order is placed, making post-order modifications impractical and operationally risky.
