- Started the register endpoint configurations:
```python
# models.py:
class CustomUser(AbstractUser):
    phone_number = models.CharField(max_length=20) 

# serializers.py:
class RegisterSerializer(serializers.ModelSerializer):
    password = serializers.CharField(write_only=True, validators=[validate_password])

    class Meta:
        model = CustomUser
        fields = ["username", "email", "phone_number", "password"]

    def validate_username(self, value):
        if len(value) < 3:
            raise serializers.ValidationError("Username muito curto.")
        return value
 
    def create(self, validated_data): 
        user = CustomUser.objects.create_user(
            username=validated_data["username"],
            email=validated_data["email"],
            phone_number=validated_data["phone_number"],
            password=validated_data["password"]
        )

        return user

# views.py:
class RegisterViewSet(generics.CreateAPIView):
    permission_classes = [AllowAny]
    serializer_class = RegisterSerializer

# urls.py:
urlpatterns = [
    path("api/register/", RegisterViewSet.as_view(), name="api-register"),
]
```

### Key Decisions
- **Why use only a 'register' endpoint and not an users endpoint?:**
    Because the register endpoint is focused on POST method only. I dont need an endpoint that makes every CRUD operation for users yet. If the project were an Social Media app, it would be necessary to see other people's profile pages. Maybe I can change this configuration to allow users to delete their accounts, but it isn't necessary now.

- **Why create the user from the '/register' path inside the serializer and not in another place of the application:**
    Because serializer doesn't only work as a translator between the database and Python. It can also validate data. When the POST method arrives, it validates the USER input, cleans it and return the validated data. Working this way is better for user input.

- **Why use AllowAny on the register viewset:**
    Because the user is not created yet. The anonymous user is creating an account so that he can, eventually, place an order. He doesn't have a token for authentication and must have access to the endpoint.

- **Why use create_user() instead of objects.create():**
    Because Django's built-in authentication system expects passwords to be hashed. The create_user() helper handles password hashing and integrates correctly with Django auth.

### Challenges
1. What happens when you create an object inside the serializer:
   - **How I solved it:** After some research on the Django REST Framework documentation, I found out that serializers have some built-in functions that you can overwrite, like create() or update()
   - **What I learned:** I learned about validated_data(). The create() function inside the serializer can receive as parameters 'self' and 'validated_data'. The last one is a dictionary that the serializer creates after cleaning the data received from the POST method.

2. I was skeptical about building validation logic on model level for the 'menu' app and going the other way on the 'users' app by building the logic inside the register serializer. Would it be inconsistent?
    - **How I solved it:** Since this problem was more about good pratices when coding, I asked a LLM about this, sending the Category model from apps.menu.models and the RegisterSerializer from apps.users.serializers for context. 
    - **What I learned:** I learned that a lot of Django applications work with validation on both levels, each one being better for each situation. Using the register serializer as an example, its better to validate its attributes because the data comes from an user input (register-form from register.html). For model-level valdiation, its more about universal rules for validation, since the application is going to use the data indefinitely.
