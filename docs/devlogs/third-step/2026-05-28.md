## 28/05/2026 - [ORDERS] Implement delivery address snapshot system
### Goal
Ensure each order permanently stores delivery address information at the time of purchase, preserving historical integrity even if the user edits or deletes their saved addresses later.

- Built a function to feed the fields
```python
def populate_delivery_fields(self):
    '''
    populates the fields related to the delivery address with information about the user's address 
        - for each field from address, it snapshots its values to keep the historical data intact
        - if the user changes the address, the database will keep the original values 
        - data-driven architecture for possible future analysis  
    '''
    self.delivery_cep = self.address.cep
    self.delivery_street = self.address.street
    self.delivery_number = self.address.number
    self.delivery_complement = self.address.complement
    self.delivery_neighborhood = self.address.neighborhood
    self.delivery_city = self.address.city
    self.delivery_state = self.address.state
    self.delivery_label = self.address.label
```

```python
def confirm_order(self):
    if self.status != self.Status.PENDING:
        raise ValidationError("Only pending orders can be confirmed.")
    
    if not self.items.exists():
        raise ValidationError("Cannot confirm an order without items.")
    
    if (self.delivery_type == self.DeliveryType.DELIVERY and self.address is None): # if the order is for delivery, it must have an address
        raise ValidationError("Delivery orders require an address.")
    
    if self.total_price < 30:
        raise ValidationError("The total price should be R$30,00 or more.")
    
    if self.delivery_type == self.DeliveryType.DELIVERY: # depending on the delivery type, the delivery fields can be null
        self.populate_delivery_fields()
    self._transition_to(self.Status.CONFIRMED)
```

### Key Decisions
- **Why create multiple order attributes related to the delivery address:**
As a business owner, the thing that I like to analyse the most is the order. Everything I do is related to how many orders I had in the week or in the day, how many I had at a certain time of the day for the last 30 days, how much I won, how many cookies I sold, everything goes back to the order table in the database. Because of that, I want to keep the history of all orders that I have on this system intact. If I only use a Foreign Key relationship of Address in the Order model, if the user changes their address, it would change the whole database too. I don't want that. So I built a logic to snapshot the delivery data at the moment of the order and kept it inside the function populate_delivery_fields().

- **Why don't use on_delete=PROTECT for OrderItem but use in Address:**
Basically, for the same reason. Even though I know the value of having data about unfinished orders to eventually find out why I am having cart problems, for example, I don't really care for it that much as the business owner. Its an easy fix for the future, but the only thing in my plans is keeping as much (finished) order data as possible.
