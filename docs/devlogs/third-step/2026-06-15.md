- Create serializers: OrderCreate
```python
class OrderCreateSerializer(serializers.Serializer):
    items = OrderItemInputSerializer(many=True)
    
    def validate_items(self, items):
        for item in items:
            try:
                menu_item = MenuItem.objects.get(pk=item["menu_item_id"])

                # checks for availability
                if not menu_item.is_available:
                    raise serializers.ValidationError(f"{menu_item.name} is not available.")
                
                # checks if the item exists
            except MenuItem.DoesNotExist:
                raise serializers.ValidationError(f"Menu item {item['menu_item_id']} does not exist.")

            try:
                stock = Stock.objects.get(menu_item=menu_item)
            
                # check the stock quantity
                if stock.quantity < item["quantity"]:
                    raise serializers.ValidationError(f"Only {stock.quantity} units available.")
            
            # check if the stock exists. if not, it means that the item's stock is not being tracked and is available
            except Stock.DoesNotExist:
                pass

        if not items:
            raise serializers.ValidationError("The cart cannot be empty.")

        return items
```

### Key Decisions
- **Why I chose to use serializer instead of a model serializer:**
Since the payload is more related to what the user wants and not to the whole Order model, I cannot build a ModelSerializer. All the user is inputing is which item they want and how many it is - at least for now.

- **Why use Stock.DoesNotExist = pass:**
Because of the business rule. Some items don't require strict stock quantity tracking and don't need to have a stock related to them. Mainly because the real stock - the cookies inside the freezer for example - has so many quantities that is irrelevant to define them on the system. It means that the item has unlimited stock. 

- **Why make the validations inside the serializer:**
Because, in this case, the serializer is the first stop of the data coming from the user's input and keeps the view cleaner. 


### Challenges
1. My attempt to make the serializer based on the Model.
   - **How I solved it:** Just like when I was building the registering API, I forgot that the flow of the data is reversed from the flow of development. In the code, you first build the model, then the serializer, then the view. But, considering the data flow, it is first created in the view, then goes to the serializer to feed the database, based on the model. Thinking about the data, it was more logical to treat the payload as it is instead of creating a serializer based on the model, specially because it is a POST request, for the moment.
   - **What I learned:** In the future I could make a GET endpoint for orders where users could see their history. In this particular case, I would use a ModelSerializer.
