## 04/05/2026 - Menu rendering with API fetch and frontend improvement
### Goal
Build the JavaScript logic to render the menu dynamically and, to develop a solid structure, I also made some improvements on the HTML and CSS to be cleaner and to fix some language issues. For example, some HTML classes and IDs were in pt-br and other were in English since I made the main HTML structure months ago. I wanted to fix this.

### Implemented
- Rendering JavaScript function
```javascript
function renderMenu(data){
    const container = document.getElementById("menu-container");

    container.innerHTML = "";

    data.forEach(category => {

        if (!category.menu_items || category.menu_items.length === 0) return;

        const categorySection = document.createElement("section");
        categorySection.classList.add("category");

        const categoryTitle = document.createElement("h2");
        categoryTitle.classList.add("category-title");
        categoryTitle.textContent = category.name;

        const itemsContainer = document.createElement("div");
        itemsContainer.classList.add("items-container");


        category.menu_items.forEach(item => {

            const itemDiv = document.createElement("div");
            itemDiv.classList.add("item");

            if (!item.is_available) {
                itemDiv.classList.add("unavailable");
            }

            itemDiv.innerHTML = `
                <img src="${item.image}" alt="${item.name}">
                <h3>${item.name}</h3>
                <p>${item.description}</p>

                <div class="price">
                    ${item.old_price ? `<span class="old-price">R$${item.old_price}</span>` : ""}
                    <span class="current-price">R$${item.price}</span>
                </div>

                <button 
                    class="botao-carrinho"
                    ${!item.is_available ? "disabled" : ""}
                    onclick="adicionarAoCarrinho('${item.name}', ${item.price})"
                >
                    ${item.is_available ? "Adicionar ao carrinho" : "Indisponível"}
                </button>
            `;

            itemsContainer.appendChild(itemDiv);
        });

        categorySection.appendChild(categoryTitle);
        categorySection.appendChild(itemsContainer);
        container.appendChild(categorySection);
    });
}
```

### Key Decisions
- **Deleted the login() function:**
    Guiding myself by common pratice and my business rules, I changed the permission_classes on the MenuItemByCategory view to allow any user - authenticated or not - to see the brand's menu and what we have to offer.

- **Separation between data fetching and UI rendering:**
    I structured the frontend logic into distinct layers: apiFetch for data retrieval, renderMenu for DOM manipulation, and init for orchestration. This makes the code more maintainable and closer to real-world frontend architecture.

- **Why I chose JS-driven rendering over Django template rendering:** 
    Since the menu data comes from a REST API, rendering it server-side with Django templates would couple the frontend too tightly to the backend view layer. By keeping the HTML shell empty and populating it via JavaScript fetch, the frontend stays decoupled and the same API can serve a mobile client in the future.

- **Handling old_price as optional data:**
    Since not every item has a promotional price, I treated old_price as an optional field on the frontend. Instead of assuming it always exists, I used a conditional render — if the field is present, the old price is displayed with a strikethrough; if not, nothing is rendered. This avoids empty or broken UI elements for regular-priced items and reflects how real product data behaves in production.

- **Handling is_available in two layers:**
    I applied the unavailability state in two places intentionally - adding an unavailable CSS class to the item card and disabling the button with the disabled attribute. The class handles the visual feedback (reduced opacity, greyed button), while the disabled attribute handles the functional restriction. Relying only on CSS would be a visual change with no real protection against interaction. Both layers together make the behavior consistent and honest to the user.

- **Skipping empty categories:**
    I decided not to render categories without items to avoid cluttering the UI with empty sections. This keeps the interface cleaner and focuses the user on available products.
