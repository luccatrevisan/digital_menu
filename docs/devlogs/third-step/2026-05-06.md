## 06/05/2026 - Implement authentication flow
### Goal
Connect the login form to backend authentication, enabling token-based requests from the frontend.

### Implemented
- Reused the login() function that I made a few days ago with a few ajustments:
```javascript
async function login(url, username, password){
    try{
        const response = await fetch(url, {
            method : "POST",
            headers : {
                "Content-Type" : "application/json",
                "X-CSRFToken": getCSRFToken() // added the CSRF token for safety, since Django uses it
            },
            body: JSON.stringify({
                username: username,
                password: password,
            })
        });

        if (!response.ok){
            if (response.status === 401){ // added error treatment for invalid login credentials
                throw new Error("invalid_credentials");
            }
            throw new Error("server_error");
        }

        const data = await response.json();
        localStorage.setItem("access", data.access); // instead of returning the token, I saved it on localStorage
        return true;

    } catch(error){
        console.error(error);
        throw error;
    }
}
```
 
- To keep the project clean, decided to create a 'user.js' file:
```javascript
const urlToken = "http://127.0.0.1:8000/api/token/";

const loginForm = document.getElementById("login-form");

loginForm.addEventListener("submit", async function(event){
    event.preventDefault();

    const username = document.getElementById("iuser").value;
    const password = document.getElementById("ipassword").value;

    try {
        await login(urlToken, username, password);
        window.location.href = ""; // redirects the user to the index page after succesful login
    } 
    catch(error) {
        const errorContainer = document.getElementById("login-error");

        if (error.message == "invalid_credentials") { 
            errorContainer.textContent = "Usuário ou senha inválidos"; // error message for invalid credentials
        } else {
            errorContainer.textContent = "Erro ao fazer login. Tente novamente."; // error message for server error
        } // both error messages appear right under the submit button
    }
});

function getCSRFToken() {
    return document.cookie
        .split('; ')
        .find(row => row.startsWith('csrftoken'))
        ?.split('=')[1];
}
```

- Created a p tag for error messages on login.html:
```html
<!-- this tag comes right after the submit input -->
<p id="login-error" style="color: red;"></p> 
```
 
- on the 'main.js' file:
```javascript
async function apiFetch(url){
    try{
        const token = localStorage.getItem("access"); // get the access token from localStorage

        const response = await fetch(url, {
            method : "GET",
            headers : {
                "Content-Type" : "application/json",
                "Authorization" : `Bearer ${token}` // passes it into the GET request method. this way, I can use the apiFetch function to GET other endpoints that will require JWT authentication.
            }
        });
    }
}
```
 
### Key Decisions
- **Why I created a separate user.js file:**
    Keeping it in main.js would mix two unrelated responsibilities. user.js owns everything related to authentication on the frontend.

- **Why I saved the token in localStorage instead of returning it:**
    In the previous version, login() returned the token directly. This meant the caller was responsible for storing and managing it. By saving to localStorage inside login(), the token management stays in one place and apiFetch() can retrieve it independently without any coupling to the login flow.

- **Why I handled authentication errors with named error strings instead of status codes:**
    Inside the form event listener, the catch block doesn't have access to the HTTP response, only to the error object thrown by login(). By throwing errors with specific messages like "invalid_credentials" and "server_error", I created a clean contract between the two layers: login() handles HTTP, user.js handles UI. This avoids leaking HTTP logic into the presentation layer.

### Challenges
1. Avoid the user's login credentials to appear on the browser URL
    - **How I solved it:** Initially, I tried to set the method attribute on the <form> tag to "POST". First, it worked, but eventually I had some problems with the error treatment that were related to the method so I removed it. The real solution was using event.preventDefault(). This stops the browser's default form submission behavior, which would send credentials as URL parameters. This way the form submission is handled entirely by JavaScript: Credentials go in the request body via JSON.stringify(), instead of going in the URL.
    - **What I learned:** HTML forms have default behavior that conflicts with JavaScript-driven flows. preventDefault() is essential when you want full control over how and where the data is sent. Credentials in a request body are not visible in the URL or browser history - which is the correct approach for authentication.

2. Understand how I should use the JWT token
    - **How I solved it:** Despite building the login() function to get the JWT access token again, I still didn't know what I should use the token for. First, I thought that I could use it to hide the cart modal that I built in August 2025 from any user that wasn't authenticated. But, eventually, I realized that it would be redundant, since I am going to develop the Cart and the Order APIs. The login function - and the access token - will be used to protect those endpoints.
    - **What I learned:** For the food business, its better to show the menu to the user and even to let him start putting his order. But, eventually, to finish the order they must be logged in or register themselves. This is the start of the logic behind the Cart & Order endpoints - that will be using the token.

3. I suffered a bit to make the error treatment work, specially on the UI.
    - **How I solved it:** Turns out I just forgot to put the script tag on the bottom of the login.html file.
    - **What I learned:** Despite the solution being simple, I made a lot of turns to find out how to solve this problem and learned some things: 
        - That I forgot to throw the error (catch block of the login() function)
        - and that the form eventListener could not catch the error because of that. This way, the error message would not appear because there would not be an error to show.

4. Had some problems with redirecting the user to index.html after the successful login
    - **How I solved it:** The login function were not returning anything, so I added 'return true' after the localStorage save. Since the form event listener waits the login function to finish before redirecting, it solved after setting a return.
    - **What I learned:** I learned that using method="POST" on the form tag would be redundant because of the javascript logic. Since the response from login() already has an 'method : "POST"' parameter, its not necessary to use the method attribute on the HTML tag.
