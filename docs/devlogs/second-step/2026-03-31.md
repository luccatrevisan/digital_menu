- Mudança das funções normalize_name e normalize_description, que estavam alterando uma instância de objeto inteira ao invés de apenas um atributo do objeto
```python
# esse código estava no antigo arquivo 'menu/validators.py'
# agora chamado de utils.py, já que não são validações

def normalize_name(value):
    splited_name = value.strip().split(" ") 
    # antes, estava usando self.name no lugar de value e passando self como parâmetro da função.
    normalized_name = []

    for word in splited_name:
        if len(word) > 3:
            normalized_name.append(word.capitalize())
        else:
            normalized_name.append(word)
    value = " ".join(normalized_name)
    return value


def normalize_description(value):
    splited_description = value.strip().split(".")
    normalized_description = []

    for phrase in splited_description:
        normalized_description.append(phrase.strip().capitalize())
    value = ". ".join(normalized_description)
    return value
```

- Continuação da limpeza baseada nos SOLID
```python
# em 'menu/models.py':

class Category(models.Model):
    def clean(self):
        self.name = normalize_name(self.name) # aqui passo como parâmetro o atributo direto. sem esperar a criação do objeto
        if self.description:
            self.description = normalize_description(self.description)
```

### Decisões
- **Por que escolhi drf-spectacular:**
  Comecei estudando um curso de Swagger porque na minha concepção era algo mais complexo. Vi como o código era feito em .yaml e acompanhei - em tutorial - a criação de uma API de consultório. Como achei algo um pouco maçante, fui pesquisar pra ver se não tiham jeitos melhores - e oficiais - de montar uma documentação.

  Pesquisando, vi que realmente as documentações não são montadas no código do Swagger direto. Muitos frameworks utilizam soluções próprias, como o FastAPI, e o Django possui soluções também. Acabei encontrando essa pela recomendação de um usuário do reddit na comunidade r/django. Depois de ler a documentação, achei bem simples e intuitivo e, por isso, optei por ela.

- **Por que não comecei o Stage 2 imediatamente:**
  Como o Django já possui um model User "embutido", preciso pesquisar e entender melhor como ele funciona. Já entendi que eu posso herdar AbstractUser, mas não ia ter muito tempo para entender detalhes a mais e eu ainda tenho que modelar a classe para usuários.

- **Por que alterei o prefetch_related de MenuItemByCategory:**
  Como é uma requisição mais extensa e profunda, o prefetch_related de antes era menos interessante, já que era baseado apenas em menuitem_set. O MenuItemByCategory é, em resumo, o cardápio completo, com todas as informações: Categorias, seus respectivos Itens, seus respectivos Estoques, seus respectivos Grupos de Complemento e seus respectivos Complementos.

  Desse jeito, o Django pega todos os itens em 4 queries SQL principais, independentemente do número de itens. O processo deixa de buscar 'um por um' por ID e passa a usar o operador IN do SQL. Primeiro, buscando todas as categorias, depois, busca todos os itens que pertencem a qualquer uma daquelas categorias de uma vez só, e segue essa lógica de 'lote' para os estoques, grupos e complementos, finalizando a estrutura dos dados em JSON.
