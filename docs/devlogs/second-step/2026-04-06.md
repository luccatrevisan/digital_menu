## 06/04/2026 - Organização final do Swagger, Encoding e Regras de negócio

### Objetivo do dia
Como hoje é a segunda-feira imediatamente após a Páscoa, preferi começar a rotina de um jeito mais leve. Passei o final de semana inteiro entregando ovos de páscoa e isso drena a saúde mental de qualquer um. Misericórdia.

O principal objetivo, portanto, era organizar a documentação da API. Como comentei no último DEVLOG, utilizei drf-spectacular para montar o swagger de um jeito mais eficiente e, hoje, fui conferir se estava tudo certo.

Encontrei alguns problemas durante isso:
- Erro de acentuação nos dados (por exemplo: "o queridinho de 2023 est� de volta! nosso cookie red velvet � a explos�o..")
- Alguns MenuItems estavam com descrições incompletas
- O Swagger estava ignorando o MinValueValidator do model ComplementGroup

### O que implementei

- default em models.py:
```python
class ComplementGroup(models.Model): # Adicionei default às quantidades (min e max) para que o Swagger use como base
    min_quantity = models.IntegerField(default=0, blank=False, null=False, validators=[MinValueValidator(0)])
    max_quantity = models.IntegerField(default=1, blank=False, null=False, validators=[MinValueValidator(1)])
```

- max_lenght em models.py
```python
class MenuItem(models.Model): 
    description = models.TextField(max_length=600, blank=False, null=False) # Aumentei o max_length de description para caber as descrições completas
```

### Decisões
- **Escolhi refazer cada item do banco de dados**
  Justamente por ser um cardápio simples e enxuto, por ter visto que algumas descrições estavam incompletas e por meu arquivo backup.json estar corrompido com esse problema de acentuação com "�", preferi atualizar os dados à mão. Assim eu pude atualizar o cardápio para as informações originais sem problema de encoding (Garanti isso no PostgreSQL com o código abaixo)

  ```SQL
    SELECT pg_encoding_to_char(encoding) FROM pg_database 
    WHERE datname = 'nome_do_banco';
  ```

### Desafios
1. Consertar o problema de acentuação/encoding de um jeito que eu não precisaria mudar manualmente. Fazer essa mudança manual nunca é uma solução interessante pensando em escala. Se eu tivesse um código maior, isso seria muito demorado e chato.
   - **Como resolvi:** Para a maioria dos scripts, utilizei o ChatGPT e o Google Gemini como base para as soluções. Passei meus dados de backup.json (que criei na mudança de SQLite3 para PostgreSQL - isso está registrado em um DEVLOG) e pedi para criarem um script que corrija o arquivo e salve no banco. Em nenhuma alternativa obtive sucesso.
   - **O que aprendi:** Eventualmente, percebi os problemas mais voltados para regra de negócio, como escolher 300 caracteres para o max_length de description (MenuItem). Ficou pouca coisa para alguns itens, como o Red Velvet unitário, mas o suficiente para outros. De qualquer forma, dobrei o tamanho para garantir.

   Percebi que o problema originou na mudança entre os bancos de dados. Vou reler o DEVLOG desse dia e vou ver o que poderia ter feito diferente para evitar esse problema. Acredito que o problema tenha ocorrido durante a exportação ou edição do arquivo backup.json.
