## 20/04/2026 - Testes no Django

### Objetivo do dia
Começar a implementação de testes e definir como será a estrutura e a organização das pastas e dos arquivos no projeto.

### O que implementei
- Instalei os pacotes
```bash
pip install pytest
pip install pytest-django
```

- Criei um arquivo pytest.ini
```python
[pytest]
DJANGO_SETTINGS_MODULE = setup.settings
```

- Criei um arquivo test_category 
```python
import pytest
from apps.menu.models import Category


@pytest.fixture
def category(db):
    return Category.objects.create(name="Pedacinho do Céu")


def test_category(category):
    assert category.name == "Pedacinho do Céu"
```

### Decisões
- **Como mantive a estrutura de testes:**
    Priorizei criar uma pasta "tests" dentro de cada app do projeto. Apesar de achar que o projeto não seja tão complexo assim pra exigir tantos testes, acho que assim ficaria mais organizado.

- **Por que não mantive a estrutura inicial do Django:**
    Ao começar o dia, o meu instinto inicial foi começar pela documentação do próprio Django apesar de ter recomendações de utilizar pytest para todos os lados da internet. Qualquer pesquisa de testes em python era sobre pytest. Porém, como o Django inicializa os apps (startapp) com um arquivo tests.py automaticamente, fui atrás de entender mais sobre os testes do framework.

    Descobri durante minhas pesquisas que o Django utiliza unittest, um pacote do Python que é baseado em classes. Logo de cara me pareceu um pouco complexo em comparação com pytest, que eu já tinha pesquisado um pouco e visto um vídeo sobre.

    A partir daí, comecei a perceber com meus próprios olhos porque pytest estava sendo recomendado pela internet inteira e, portanto, preferi criar uma estrutura de testes baseada nele ao invés de utilizar o que o Django já oferece.

- **Por que usar pytest e não unittest:**
    Pytest permite desenvolvimento de testes mais curtos e legíveis, baseados em funções e com recursos como fixtures que simplificam a criação e reutilização de dados. Em contrapartida, o unittest é baseado em classes e pode tornar os testes mais verbosos.

### Desafios
1. Não soube exatamente o que testar.
   - **Como resolvi:** Como eu não tinha muito essa "cultura" de testes, precisei parar e pensar um pouco no motivo de fazer testes. É até por isso que hoje foi um dia mais focado em aprendizado do que código em si.
   - **O que aprendi:** Aprendi que testes são principalmente voltados para partes importantes do código, como as requisições e seus códigos, regras de negócio, entre outras. Basicamente, o pensamento é simples: "Se essa parte poder quebrar o código, preciso de testes e várias ocasiões em que ele possa dar erro".
