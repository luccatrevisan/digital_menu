## 07/04/2026 - Models de usuário e de endereços

### Objetivo do dia
Aprender a fazer uso do model de User já embutido no Django para acelerar o desenvolvimento e começar a criar o meu próprio usuário, com número de celular e endereço.

### O que implementei
- Model CustomUser
```python
class CustomUser(AbstractUser):
    phone_number = models.CharField(max_length=20) # simples e direto, porque uso nome e email do próprio django.
```

- Model Address
```python
class Address(models.Model):
    LABEL_CHOICES = [
        ('CASA', 'Casa'),
        ('TRABALHO', 'Trabalho')
    ]

    user = models.ForeignKey(CustomUser, on_delete=models.CASCADE, related_name='addresses')
    cep = models.CharField(max_length=9)
    street = models.CharField(max_length=255)
    number = models.CharField(max_length=20)
    complement = models.CharField(max_length=255, blank=True) 
    neighborhood = models.CharField(max_length=100)
    city = models.CharField(max_length=100)
    state = models.CharField(max_length=2)
    label = models.CharField(max_length=50, choices=LABEL_CHOICES) 

    def __str__(self):
        return f'{self.street}, {self.number} - {self.city}/{self.state}'
```

- Configurações em settings.py
```python
INSTALLED_APPS = [
    'apps.users.apps.UsersConfig',
]

AUTH_USER_MODEL = 'users.CustomUser'
```

- apps.users.admin:
```python
class AddressAdmin(admin.ModelAdmin):
    list_display = ['user', 'street', 'number']

admin.site.register(Address, AddressAdmin)
admin.site.register(CustomUser)
```

### Decisões
- **Porque criar model de endereços:**
  Preferi criar um model específico já que um usuário pode ter vários endereços cadastrados e endereços por si só tem muitos atributos.

- **Porque não usar relacionamento Many To Many entre CustomUser e Address:**
  Logo de cara, é fácil dizer que uma pessoa pode pedir para diferentes endereços. Já atendi clientes na Chewie Cookies que pediram em um bairro, se mudaram e pediram nesse novo lugar ainda pelo mesmo sistema de cardápio online. Logo, um CustomUser teria vários endereços mas um endereço pertenceria somente a esse usuário.

  Porém, fui além. Pensei numa situação específica, como já aconteceu comigo: Fui pedir um lanche por um app de delivery com um endereço e minha mãe pediu na mesma loja, no mesmo app, em outro dia mas para o exato mesmo endereço - nossa casa. Nesse caso, um endereço não pertenceria a duas pessoas?

  O que me fez decidir por um relacionamento ForeignKey (1 para muitos) nessa modelagem foram:
  - O fato de minha mãe ter que preencher o formulário pelo celular dela novamente apesar de ser o mesmo endereço nesse meu caso acima;
  - O fato de um cliente aleatório não ter que escolher entre todos os endereços do mundo (ou só do Brasil, pelo menos) para decidir qual será o lugar da entrega para ele.

  Nesse último ponto, acredito que um relacionamento ManyToMany dificultaria a proteção de endereços de cada cliente, onde a seleção de endereço seria realizada com TODOS os endereços cadastrados no banco. Não tenho certeza como isso funcionaria exatamente mas imagino que exista essa questão de proteção de dados também.

- **Porque usar Charfield para número do endereço:**
  Como não é um dado com o qual vou realizar operações e ele pode conter caracteres diferentes (como "Rua X, número 21B"), preferi Charfield ao invés de Integer, onde teria problemas principalmente com esse caso de letras.

### Desafios
1. Tive problema na hora de fazer as migrations depois de criar CustomUser
   - **Como resolvi:** Limpei o schema público pelo PostgreSQL com ```sql DROP SCHEMA public CASCADE``` para apagar o histórico inconsistente de migrations. Fiquei travado por conta de comandos mais complexos em SQL e ter que lidar com o histórico de migrations do Django.
   - **O que aprendi:** Em projetos futuros, é bom começar pela modelagem do User antes de sair fazendo um monte de alteração e salvando no banco de dados, principalmente se forem em Django. Como ele já tem o model User, qualquer migration salva antes de customizar ele pode bloquear futuras mudanças. Definir a variável AUTH_USER_MODEL em settings.py antes da primeira migration resolve.
