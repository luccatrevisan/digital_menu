## 14/04/2026 - Implementar granular permissions

### Objetivo do dia
Depois de avançar em autenticação JWT, quis seguir para granular permissions com o objetivo de especificar quem pode apenas visualizar - e não editar - os endpoints da API.

### O que implementei
- Comecei implementando um método na View de Category
```python
def get_permissions(self):
        if self.request.method in SAFE_METHODS:
            return [AllowAny()]
        return [IsAdminUser()]
```

- Porém, vi que ia me repetir e criei o arquivo permissions.py
```python
from rest_framework.permissions import BasePermission, SAFE_METHODS

class IsAdminOrReadOnly(BasePermission):
    # como as permissões são, em resumo, uma lista de classes, criei uma com a função abaixo.

    def has_permission(self, request, view):
        # verifica se o método da requisição não permite edição (safe, segura)
            # se for, qualquer um tem permissão
            # se não for, apenas administradores e usuários de staff tem permissão
        if request.method in SAFE_METHODS:
            return True
        return request.user and request.user.is_staff
```

- Importei essa classe para views.py
```python
from apps.menu.permissions import IsAdminOrReadOnly
```

- Adicionei a permissão em cada viewset
```python
class CategoryViewSet(viewsets.ModelViewSet):
    authentication_classes = [JWTAuthentication]
    permission_classes = [IsAuthenticated, IsAdminOrReadOnly]
```

### Decisões
- **Por que escolhi criar a minha própria classe de permissão granular:**
    Inicialmente, quando estava pesquisando sobre as granular permissions, pensei em usar as permissões simples, já embutidas no framework, como AllowAny() e IsAdminUser(), dentro da variável "permission_classes" nas views do projeto. 
    
    Porém, como eu acho mais seguro variar a permissão de acordo com o método da requisição, me vi na necessidade de criar funções que verificam o método.
    
    Depois, vi que ia repetir a função em todas as views e, para não fazer código repetitivo, optei por criar a lógica em permissions.py

### Desafios
1. Fiquei em dúvida como utilizar as permissões para views de serializers aninhados
   - **Como resolvi:** Fiquei pensando se não seria redundante definir permissões granulares em ComplementByGroup ou em MenuItemByCategory, que são endpoints aninhados, considerando que o Django poderia associar as permissões de cada serializer ao aninhado. Por exemplo, em ComplementByGroup, como a view de Complement e a view de ComplementGroup já possuíam suas permissões definidas, o aninhado "herdaria" essas permissões. Porém, descobri que isso não acontece pesquisando no google: O DRF valida as permissões por view, não pelo serializer, o que significa que uma view de um serializer aninhado seria considerado uma outra view isolada.
   - **O que aprendi:** Como as views dos serializers aninhados são apenas Read Only, não existe a preocupação de definir permissões granulares para elas. Na própria documentação do swagger, os endpoints aninhados da API possuem apenas um método: GET.
